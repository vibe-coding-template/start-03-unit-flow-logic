### 1. 範本倉庫 (Template Repo) 配置建議
控制流作業的核心在於「時序精準度、狀態一致性與安全防護」，範本應包含指令流偵錯工具與事件監控框架，建議包含：
*   **📂 `index.html`**：**指令流可視化監控器**。預設包含一個「事件日誌 (Event Logger)」，學員在開發時能即時看到 `setInterval` 觸發的頻率與具體送出的指令。
*   **📂 `logic.js`**：**核心控制邏輯區**。學員在此實作定時器啟動/停止、多鍵狀態聚合與安全監聽器。預留 `// TODO` 引導學員處理定時器記憶體洩漏問題。
*   **📂 `README.md`**：**系統穩定性分析錄**。提供模板要求學員紀錄：在不同的 `interval`（如 10ms vs 100ms）下，機器人的反應與穩定性有何差異？
*   **📂 `tests/`**：**邏輯邊界測試**。使用 Actions 或腳本自動驗證：當視窗失去焦點 (Blur) 時，系統是否真的送出了 `STOP` 指令？

### 2. 作業任務部署細節

#### 任務 1：指令發送「甜蜜點」實驗 (Interval Tuning Lab)
*   **目標**：尋找效率與穩定的平衡點。掌握非同步迴圈的正確生命週期管理（預防重複啟動定時器）。
*   **Classroom 部署建議**：
    *   **定時器實作挑戰**：撰寫一個穩健的 `startFlow` 函式，確保在連發過程中，不會因為重複點擊而產生定時器堆疊。
    *   **頻率測試實驗**：在 README 中詳細紀錄 10ms、50ms、100ms 下的操作感，並說明為什麼過快的頻率（如 1ms）會導致硬體端潰散。
    *   **驗證要點**：**資源管理審核**。學員是否在啟動前檢查了 `flowInterval` 是否已存在？選定的頻率是否能兼顧順暢與安全？

#### 任務 2：多按鍵同時輸入衝突解決 (Multi-Key Logic Lab)
*   **目標**：實作「平行指令處理」。確保當使用者同時按下「前進」與「左轉」時，系統能產生合併指令，而非讓機器人在兩者間跳動或鎖死。
*   **Classroom 部署建議**：
    *   **狀態機設計實作**：使用一個物件（如 `keysActive`）記錄所有按鍵的真假值，並在每個循環中根據該物件內容合成最終 Payload。
    *   **複雜狀態驗證**：模擬快速交替按鍵的操作，驗證是否有按鍵被「黏死」或狀態未正確重置的現象。
    *   **驗證要點**：組合指令的輸出是否符合物理直覺？代碼邏輯是否清晰且易於擴展到更多按鍵？

#### 任務 3：斷訊安全停機機制 (Dead Man's Switch Lab)
*   **目標**：實作「失效安全」。確保在極端狀況下（網頁當機、切換分頁、用戶分心），機器人具備自動歸零、防衝撞的自我保護能力。
*   **Classroom 部署建議**：
    *   **危險事件監聽實作**：監聽 `window.blur` 或 `visibilitychange` 事件，並在觸發時立即執行 `stopFlow` 並送出 `STOP` 命令。
    *   **模擬失控測試**：在 README 中錄製或截圖證明：當按住按鈕時切換到別的視窗，系統日誌是否顯示了自動停機。
    *   **驗證要點**：安全機制的優先權是否足夠高？是否徹底清除了所有掛載的定時器以釋放系統資源？

### 3. 時序健壯性與安全意識點評標準 (Temporal Robustness & Safety Standards)
此單元的價值在於 **「對非同步流程的絕對掌控力與對防護邏輯的工程嚴謹力」**：
*   [ ] **資源釋放嚴謹度**：定時器是否在所有預期路徑（放開按鍵、安全停機）中都被正確 `clearInterval`？
*   [ ] **狀態聚合準確性**：多鍵並發時，生成的指令是否能精準反映最後一個綜合的操作意圖？
*   [ ] **安全防護覆蓋率**：除了正常的 `mouseup`，是否考慮到了滑鼠滑出視窗、視窗失去焦點等異常邊界情境？
